Technology brief · Vespa.ai

One engine instead of three. Vespa unifies keyword search, vector search, and ML ranking — and runs the ranking next to the data.

The problem today

🔤 Keyword engine
Elasticsearch / OpenSearch
🧭 Vector database
Pinecone / Weaviate
🎯 Re-ranking service
a separate model server
+ glue code to fan out & merge
→
Vespa.ai
Find · Match by meaning · Rank
all in a single query, where the data lives
keyword (BM25) vectors (ANN) filters ML ranking

The pain of three systems: two copies of the data · ranking happens far from the data (network hop) · updates lag · lots to operate.

What it is

An open-source engine that stores your data and does full-text + vector + structured search with machine-learned ranking in one system — computing the ranking on the same nodes that hold the data.

Why trust it

  • Built inside Yahoo (~150 apps, ~800k queries/sec, ~1B users). Open-sourced 2017; independent company since 2023.
  • Powers Perplexity’s retrieval, Spotify podcast search, Vinted marketplace.
Factored · Vespa upskilling · for TTO review
Why it matters · and what I built this weekend

The capability — and the proof.

What makes it different

  • True hybrid in one query — keyword + vector + filters, fused in ranking (not merged in app code).
  • ML ranking next to the data — hundreds of signals, multi-phase, no network hop to a model server.
  • Real-time updates — a price or a click changes ranking on the next query, no reindex.
  • Scales to billions of docs/vectors — one system to operate, not three.

Where it fits — ezCater already runs Vespa

ezCater's search platform is Vespa (Go · Temporal · Kafka), and they're actively building out relevance & ranking. The value-add isn't a new engine — it's Vespa fluency on their domain: hybrid semantic discovery, ML ranking, the mixed-dietary problem. I built a working catering-discovery demo to prove it.

I built it, and measured it

hands-on · ran it locally Deployed Vespa in Docker, fed 171,332 documents (Vespa embedded each one itself), and ranked queries three ways. Quality measured with nDCG@10 on the full index:

Hybrid (RRF) 0.74
Keyword (BM25)0.59
Semantic only0.51

Hybrid wins by +25% — it catches the exact term and the meaning. 171k docs, queries in 5–14 ms, all on one laptop — and it scales horizontally to billions.

Honest trade-offs

⚠ Steeper learning curve (YQL, schemas, ranking) · heavier ops than a hosted vector DB for tiny jobs · custom logic is Java/JVM · smaller talent pool. Budget weeks, not days, to production.

Hook: “Most teams that build search or RAG run three separate systems: a keyword engine like Elasticsearch, a vector database like Pinecone, and a separate service to re-rank with a model. That means two copies of the data, ranking that happens across the network far from the data, and a freshness lag — plus a lot of glue.”

Its key idea: it runs the ranking and the ML models on the same nodes that store the data — so keyword, vector, filters, and a learned model are evaluated together, in one query.”
